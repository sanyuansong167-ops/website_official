# 官网可视化编辑模式：后端接口缺口清单

> 版本：v1.0  
> 更新日期：2026-07-14  
> 适用范围：官网 Portal、Admin 后台、页面构建器、产品、案例、行业方案。  
> 关联文档：`docs/接口文档.md`、`docs/重构接口文档.md`、`docs/数据库设计说明.md`、`docs/错误码说明.md`、`plan/官网可视化编辑模式执行计划.md`。

## 1. 结论

现有后端已经具备页面构建器的部分核心能力，包括受控组件模板、页面定义、页面草稿、受控预览 Token、页面发布、版本历史和回滚；它可以作为可视化页面编辑的基础。

但现有接口**不能直接满足**本期“官网全站可视化编辑模式”的完整要求，原因如下：

1. 当前接口没有服务端独占编辑锁，不能满足“同一资源不允许多个管理员同时修改”。
2. 现有传统业务模块多数为直接 CRUD，写入后即改变 Portal 可读数据，不能满足统一的“草稿—预览—发布”流程。
3. 产品、案例、行业方案缺少独立详情编辑器所需的草稿、预览、发布快照、版本、回滚和编辑锁接口。
4. 页面发布与详情内容发布之间的引用校验、缓存失效、下线影响处理尚未形成完整接口契约。
5. 预览读取接口目前以 Token 作为访问凭据；若产品要求“普通用户即使获得链接也不可预览”，还需增强预览访问鉴权。

本清单只描述为满足既定执行计划必须补齐或调整的后端能力；不要求首期开放任意 HTML、JavaScript、CSS 或未注册组件的编辑能力。

### 2.1 已有能力

| 能力 | 现有接口/文档位置 | 判定 |
| --- | --- | --- |

## 4. GAP-01：服务端独占编辑锁

### 4.1 现状与问题

现有写接口主要依赖 `version` 乐观锁。乐观锁只能在提交时发现版本冲突，不能阻止第二名管理员同时进入编辑、分别修改本地草稿并反复冲突，不满足本期明确要求的单人编辑。

页面构建器、产品、案例、行业方案均未在接口文档中定义创建锁、续租心跳、主动释放、查询锁状态和强制解锁能力。

### 4.2 数据模型要求

新增统一编辑锁表或等价 Redis + 持久审计模型。建议锁记录至少包含：

| 字段 | 说明 |
| --- | --- |
| `resourceType` | `PAGE`、`PRODUCT`、`CASE`、`INDUSTRY_SOLUTION`。 |
| `resourceId` | 对应资源主键。 |
| `lockToken` | 服务端生成的高熵随机令牌；不得记录到前端日志。 |
| `ownerAdminId` | 持锁管理员 ID。 |
| `ownerDisplayName` | 用于向其他管理员展示占用提示的脱敏显示名。 |
| `acquiredAt` | 获锁时间。 |
| `heartbeatAt` | 最近心跳时间。 |
| `expiresAt` | 锁到期时间。 |
| `forceUnlockedBy` | 强制解锁操作人；无则为空。 |
| `forceUnlockReason` | 强制解锁原因；强制解锁时必填。 |

唯一约束必须覆盖 `(resource_type, resource_id)` 的有效锁。锁创建、过期回收和抢占必须由服务端原子执行，不能由前端判断后再写入。

### 4.3 必须新增的统一接口

以下路径可统一采用资源型设计；最终命名须同步写入 `docs/接口文档.md`。

| 能力 | 方法与路径建议 | 请求要点 | 成功响应要点 |
| --- | --- | --- | --- |
| 获取锁状态 | `GET /admin/api/editor-locks/{resourceType}/{resourceId}` | 资源类型与 ID | 是否可编辑、持有人、到期时间；不得返回完整 `lockToken`。 |
| 创建/获取锁 | `POST /admin/api/editor-locks/{resourceType}/{resourceId}` | 可选编辑会话备注 | `lockToken`、`expiresAt`、`heartbeatIntervalSeconds`。 |
| 锁心跳 | `POST /admin/api/editor-locks/{resourceType}/{resourceId}/heartbeat` | `lockToken` | 新的 `expiresAt`。 |
| 主动释放 | `DELETE /admin/api/editor-locks/{resourceType}/{resourceId}` | `lockToken` | 幂等成功。 |
| 强制解锁 | `POST /admin/api/editor-locks/{resourceType}/{resourceId}/force-unlock` | 解锁原因必填 | 解锁审计 ID、执行时间。 |

也可采用嵌套资源路径，例如 `POST /admin/api/page-builder/pages/{id}/lock`；但页面和三类详情必须遵循同一字段、错误码和行为语义。

### 4.4 强制规则

1. 创建锁必须校验管理员身份、CSRF 和资源存在性。
2. 第二名管理员申请未过期资源锁时，必须返回可行动的业务错误，包含持有人显示名和预计到期时间，但不得泄露敏感账户信息。
3. 心跳只能由锁持有人且携带正确 `lockToken` 执行；过期锁不得续期。
4. 退出编辑器时前端主动释放；浏览器异常退出由超时自动释放兜底。
5. 强制解锁只允许具备明确权限的管理员执行，且必须记录操作人、原因、原持有人和资源信息。
6. 草稿保存、预览创建、发布、回滚、删除或下线等破坏性动作必须在服务端二次校验有效锁，不能只依赖前端已获锁状态。

### 4.5 推荐错误码类别

具体数值需写入 `docs/错误码说明.md`，至少增加以下稳定业务语义：

| 语义 | HTTP 状态建议 | 返回信息要求 |
| --- | --- | --- |
| 资源已被其他管理员锁定 | `409` | 持有人显示名、到期时间、是否可申请强制解锁。 |
| 锁不存在或已过期 | `409` | 提示重新进入编辑并申请锁。 |
| 锁归属不匹配 | `403` 或 `409` | 不暴露完整锁信息。 |
| 无强制解锁权限 | `403` | 明确提示权限不足。 |
| 发布时锁已失效 | `409` | 禁止发布，提示重新获取锁并刷新草稿。 |

## 5. GAP-02：页面构建器链路的锁校验与发布前置校验

### 5.1 现有主链路

当前页面构建器已经定义：

- 组件模板：`GET /admin/api/page-builder/component-templates`；
- 草稿读取与保存：`GET/PUT /admin/api/page-builder/drafts/{pageId}`；
- 创建与撤销预览：`POST /admin/api/page-builder/drafts/{pageId}/previews`、`DELETE /admin/api/page-builder/previews/{previewToken}`；
- 发布、版本、回滚：以页面定义为中心的发布与版本接口。

这些接口足以承载受控区块和合法插槽，但当前文档未要求调用方持有有效编辑锁。

### 5.2 修改要求

| 现有接口 | 必须增加的要求 |
| --- | --- |
| `PUT /admin/api/page-builder/drafts/{pageId}` | 请求 Header 或请求体传入 `lockToken`；服务端校验 `PAGE:{pageId}` 有效锁、锁归属和草稿 `version`。 |
| `POST /admin/api/page-builder/drafts/{pageId}/previews` | 必须校验持锁状态；预览 Token 绑定草稿 `schemaHash`、创建人、资源类型和资源 ID。 |
| 页面发布接口 | 必须校验锁、草稿版本、Schema 校验结果、引用实体有效性和缓存失效计划。 |
| 页面回滚接口 | 必须校验锁；回滚不得覆盖当前草稿，应生成新的发布版本或按既定规则同步草稿。 |
| 页面删除/禁用接口 | 若存在有效锁，除持锁人或具备强制权限者外不得执行。 |

### 5.3 页面 Schema 及受控区块校验

保存草稿与发布前均必须执行以下校验，且发布不能仅信任前端已经校验过的结果：

1. `sections[].componentCode` 必须存在于已启用的组件模板白名单。
2. 区块只能放入该组件定义允许的页面插槽；同一插槽中的数量可扩充，但不得突破组件定义的数量、嵌套或业务规则。
3. 区块属性必须通过对应 Schema 校验，过滤脚本、事件属性、危险 URL 协议和未声明字段。
4. 媒体 ID 必须存在、可用、未被非法删除，且符合当前用途的媒体规则。
5. 内容引用必须指向可发布的产品、案例、行业方案或允许被预览的草稿实体；不得向 Portal 暴露草稿引用。
6. `routePath`、页面唯一 Key、SEO 字段和页面状态必须符合统一约束。

## 6. GAP-03：预览访问的权限边界

### 6.1 现状

当前预览读取路径为 `GET /portal/api/page-builder/previews/{previewToken}`，文档定义为不登录访问、使用预览 Token 作为凭据、Token 有效期为 1 小时并支持撤销。

这能避免预览出现在普通 Portal 导航中，但“知道完整 Token 的任何人”仍可读取预览。因此需要根据产品安全要求明确采用下列一种策略。

### 6.2 首选策略

首期采用“**管理员会话 + 预览 Token 双重校验**”：

1. 预览数据接口迁移到 `/admin/api/**`，或保留原路由但要求有效管理员会话。
2. Token 只用于将某次预览与指定草稿哈希和操作人绑定，不作为唯一认证凭据。
3. 预览页前端必须由 Admin 编辑器路由加载，不在公开页面输出入口、链接或 Token。
4. 响应保留 `Cache-Control: no-store`，日志、监控、错误消息和前端埋点中禁止记录完整 Token。

如业务必须允许外部审核人查看预览，则须作为独立功能设计：生成一次性、最短有效期、可设置审核人和访问次数的分享凭据；该能力不纳入本期默认交付。

### 6.3 预览接口验收

- 无管理员会话访问预览时返回未授权，不返回任何草稿内容。
- Token 过期、撤销、草稿哈希已变化或资源未发布许可不足时，统一返回不可访问结果。
- 响应中不包含 `binding`、查询配置、内部模块名、编辑锁、审计记录、完整媒体存储路径或其他后台字段。
- 预览不会写入正式 Portal 缓存，且不会污染搜索引擎索引。

## 7. GAP-04：传统模块写入的草稿化迁移

### 7.1 问题

现有首页、企业实力、AI 战略、核心能力、导航、页头页脚及其他传统内容模块的后台接口多为直接新增、更新、删除、显示状态与排序操作。按现有缓存约定，事务提交后会失效 Portal 缓存，使改动即时影响普通用户。

这与本期“保存草稿不公开、经预览后发布才公开”的规则冲突。因此，不能仅在前端增加编辑器界面，而后端仍调用传统写接口。

### 7.2 改造原则

1. 已纳入页面构建器的展示内容，优先收敛为页面 Schema 的受控区块属性或绑定配置。
2. 仍需保留独立业务表的内容，必须新增草稿表/草稿快照和发布快照，不得直接修改 Portal 读取记录。
3. 原传统 CRUD 在迁移完成前仅允许用于非门户展示数据或受明确管控的运维操作；不得成为编辑器保存、预览、发布流程的旁路。
4. 每个一级导航及辅助页面必须建立“页面路由 → 草稿来源 → 发布快照 → Portal 查询”的映射表并写入实施记录。

### 7.3 9 个一级导航的后端核对范围

| 路由 | 迁移目标 | 后端要求 |
| --- | --- | --- |
| `/` | 首页页面 Schema | 页面草稿、受控区块、发布快照。 |
| `/strength` | 企业实力页面 Schema | 原模块数据仅能经草稿/发布链路影响前台。 |
| `/ai-strategy` | AI 战略页面 Schema | 同上。 |
| `/capabilities` | 核心能力页面 Schema | 同上。 |
| `/products` | 产品列表页 Schema + 已发布产品查询 | 页面与产品发布解耦但需引用校验。 |
| `/solutions` | 行业方案列表页 Schema + 已发布行业方案查询 | 页面与方案发布解耦但需引用校验。 |
| `/cases` | 案例列表页 Schema + 已发布案例查询 | 页面与案例发布解耦但需引用校验。 |
| `/about` | 关于我们页面 Schema | 发展历程等内容需草稿化或纳入页面 Schema。 |
| `/contact` | 联系页面 Schema | 展示信息走草稿发布；线索提交接口独立保留。 |

页头、页脚、导航、移动端菜单、SEO 默认配置、404/错误页、隐私政策和服务条款等公共或辅助页面，也必须纳入对应的草稿与发布边界；不允许通过未受控接口直接改动已发布页面效果。

## 8. GAP-05 至 GAP-07：三类独立详情编辑器接口

### 8.1 统一生命周期

产品、案例、行业方案都必须具备同等生命周期：

```text
创建或打开详情草稿
  → 获取独占锁
  → 编辑基础字段、媒体、富文本、SEO、关联关系
  → 保存草稿
  → 受控预览
  → 发布快照
  → 查看版本历史
  → 回滚为新发布版本
```

现有 `/admin/api/products`、`/admin/api/cases`、`/admin/api/industry-solutions` 的 CRUD 不能替代这一生命周期。

### 8.2 必须新增的详情接口矩阵

以下以产品为例。案例和行业方案仅替换资源根路径，字段规则根据各自详情模型扩展，但流程、鉴权、错误码和锁语义必须一致。

| 能力 | 产品路径建议 | 案例/行业方案等价路径 | 关键要求 |
| --- | --- | --- | --- |
| 获取编辑详情/草稿 | `GET /admin/api/products/{id}/draft` | `/cases/{id}/draft`、`/industry-solutions/{id}/draft` | 返回当前草稿、版本、媒体、SEO、关联关系。 |
| 创建实体与初始草稿 | `POST /admin/api/products` | 等价资源路径 | 创建不等于公开发布；初始状态为草稿。 |
| 保存详情草稿 | `PUT /admin/api/products/{id}/draft` | 等价资源路径 | 必须携带 `version` 和 `lockToken`。 |
| 获取/创建编辑锁 | 统一锁接口或 `POST /admin/api/products/{id}/lock` | 等价资源路径 | 使用统一锁协议。 |
| 锁心跳/释放 | 统一锁接口或资源嵌套路径 | 等价资源路径 | 使用统一锁协议。 |
| 创建预览 | `POST /admin/api/products/{id}/draft/previews` | 等价资源路径 | 预览仅能读取当前草稿。 |
| 获取预览 | 管理员受保护预览路径 | 等价资源路径 | 不得进入 Portal 已发布数据接口。 |
| 撤销预览 | `DELETE /admin/api/products/{id}/previews/{token}` | 等价资源路径 | 幂等、可审计。 |
| 发布 | `POST /admin/api/products/{id}/publish` | 等价资源路径 | 生成不可变发布快照，失效相关缓存。 |
| 获取版本 | `GET /admin/api/products/{id}/versions` | 等价资源路径 | 返回版本摘要、发布人、时间、来源。 |
| 回滚 | `POST /admin/api/products/{id}/rollback/{versionId}` | 等价资源路径 | 校验锁，回滚生成新版本。 |
| 下线 | `POST /admin/api/products/{id}/offline` | 等价资源路径 | 校验引用影响并更新 Portal 可见性。 |

### 8.3 数据与校验要求

每类详情草稿和发布快照至少要覆盖：基础字段、摘要、封面、图片/媒体、正文富文本、SEO、可见性、排序、关联实体、状态、版本、创建与更新时间。

保存和发布时必须校验：

1. 富文本经过服务端白名单清洗；不允许脚本、危险事件属性和非法嵌入内容。
2. 外部链接只允许白名单协议与符合业务规则的域名策略。
3. 媒体必须存在、可用、满足引用类型规则；已发布版本引用的媒体不得被误删。
4. 关联产品、案例、行业方案必须存在、状态合法，并防止不允许的循环或重复关联。
5. SEO 字段、标题、状态和显示控制符合当前实体的唯一性与长度约束。
6. 删除、下线、回滚操作必须在事务内检查页面与其他实体引用关系。

### 8.4 Portal 查询边界

Portal 列表和详情接口只能读取已发布、可见且未逻辑删除的数据。草稿、预览、锁、版本、审计信息绝不能经 `/portal/api/**` 返回。

对不存在、草稿、下线、隐藏或已删除的实体，应返回统一的公开不可见结果，不能通过错误细节暴露后台状态。

## 9. GAP-08：引用关系、发布联动与缓存

### 9.1 引用关系能力

当页面区块引用产品、案例、行业方案时，后端需维护可查询的引用关系，至少支持：

| 能力 | 说明 |
| --- | --- |
| 查询被引用位置 | 管理员在详情编辑器中可查看“被哪些页面/区块引用”。 |
| 发布前校验 | 发布页面或详情前，校验目标实体可被当前状态引用。 |
| 下线/删除拦截 | 被已发布页面强引用时，按统一规则阻止、自动隐藏或要求替换；首期须明确一种规则并全局一致。 |
| 回滚影响评估 | 详情回滚到不再满足当前页面依赖的数据时，必须阻止或给出可执行修复方案。 |
| 媒体引用追踪 | 任何已发布快照引用的媒体都必须防止误删。 |

### 9.2 发布后的缓存失效

保持现有“事务提交后删除 + 延迟二次删除”规则。发布接口要在同一事务边界内计算影响范围，提交成功后失效：

1. 当前页面或详情的 Portal 缓存；
2. 对应列表缓存；
3. 引用该实体的已发布页面渲染缓存；
4. 相关 SEO/元数据缓存；
5. 受影响的关联详情缓存。

缓存删除失败不能把草稿误当作发布成功；必须记录结构化日志、监控告警和可重试任务，并由 TTL 兜底。

## 10. GAP-09：横向接口契约、审计与文档

### 10.1 鉴权与请求约束

- 所有 `/admin/api/**` 读写接口都校验登录与管理员角色；写接口必须校验 CSRF。
- 所有修改草稿、创建预览、发布、回滚、下线、删除和解锁接口都必须校验资源锁与版本。
- 统一响应体必须保留业务错误码、用户可读提示与 `traceId`；禁止直接暴露 SQL、堆栈、文件路径、密钥或完整 Token。
- `lockToken` 与 `previewToken` 不得在 URL 日志、Referer、监控标签和前端埋点中明文记录。

### 10.2 审计事件

至少记录以下事件：获取锁、心跳异常、主动释放、强制解锁、保存草稿、创建/撤销预览、发布、发布失败、回滚、下线、删除、引用校验失败和缓存失效失败。

审计记录至少包含资源类型/ID、操作人、操作时间、来源 IP 或会话标识、请求追踪 ID、操作前后版本、结果、失败原因和必要摘要；不保存完整富文本、密码、完整 Token 或敏感线索信息。

### 10.3 必须同步的工程交付物

| 交付物 | 完成标准 |
| --- | --- |
| Controller/Service/Repository | 每个新增接口有明确实现位置，事务边界和权限校验可追溯。 |
| 数据库迁移 | 锁、草稿、发布快照、版本、引用关系及必要索引均有可回滚/可验证迁移。 |
| `docs/接口文档.md` | 新增、修改接口完整记录方法、路径、鉴权、参数、响应、错误码、缓存影响和实现位置。 |
| `docs/重构接口文档.md` | 页面构建器接口的最新权威契约与旧文档一致。 |
| `docs/错误码说明.md` | 新增锁、草稿、预览、发布、引用和缓存类错误码。 |
| `docs/数据库设计说明.md` | 新增表、状态机、索引、约束、数据迁移策略。 |
| 自动化测试 | 覆盖权限、CSRF、锁、版本冲突、草稿隔离、预览、发布、回滚、缓存与引用关系。 |

## 11. 实施顺序与交付条件

### 11.1 第一批：页面编辑基础设施

**交付内容**：GAP-01、GAP-02、GAP-03 的页面部分。

**完成条件**：

- [ ] 管理员可以锁定首页页面资源，第二名管理员被明确拒绝。
- [ ] 心跳、超时释放、主动释放和授权强制解锁均可用且有审计。
- [ ] 未持有有效锁不能保存首页草稿、创建预览、发布或回滚。
- [ ] 页面预览不会向未授权普通用户泄露草稿数据。
- [ ] 页面 Schema 能阻止非法组件、插槽、脚本链接和无效媒体。

### 11.2 第二批：全站页面迁移

**交付内容**：GAP-04，覆盖 9 个一级导航、公共布局和辅助页面。

**完成条件**：

- [ ] 任何纳入可视化编辑范围的前台内容都不再通过传统 CRUD 直接公开。
- [ ] 每个页面均可以完成草稿保存、受控预览、发布、版本查询与回滚。
- [ ] 公共页头、页脚、导航、移动菜单和默认 SEO 不存在绕过发布链路的写入口。
- [ ] Portal 只能读取已发布数据，缓存与发布结果一致。

### 11.3 第三批：三类详情编辑器

**交付内容**：GAP-05、GAP-06、GAP-07、GAP-08。

**完成条件**：

- [ ] 产品、案例、行业方案均有草稿、锁、预览、发布、版本与回滚接口。
- [ ] 三类实体的独立详情编辑流程不调用会直接影响 Portal 的传统编辑接口。
- [ ] 页面绑定、关联关系、下线、删除、回滚和缓存失效场景均完成端到端测试。
- [ ] 已发布媒体引用可追踪，不能被后台误删。

### 11.4 第四批：发布治理与上线验收

**交付内容**：GAP-09 与全链路压测、安全验证、故障演练。

**完成条件**：

- [ ] 所有新增接口、错误码、表结构和缓存行为均已同步文档。
- [ ] 自动化测试覆盖正常流程、越权、CSRF、锁冲突、版本冲突、Token 失效、缓存故障和引用异常。
- [ ] Portal 包不加载编辑器资源，不请求 Admin API，不包含草稿或管理字段。
- [ ] 发布失败、缓存失败、锁失效、强制解锁均有可追踪审计和可执行恢复措施。

## 12. 非本期范围

以下能力不作为本期缺口补齐目标，避免扩大首期风险：

1. 多管理员协同编辑、多人实时光标、评论与冲突合并。
2. 移动端完整编辑体验；首期只承诺桌面端编辑。
3. 任意 HTML、JavaScript、CSS、第三方脚本或自定义未注册组件。
4. 面向外部人员的可转发预览分享、审核工作流和审批流。
5. 富文本编辑器具体选型与复杂扩展；本期只预留服务端富文本安全校验和媒体引用能力。

## 13. 最终验收标准

后端接口补齐完成的最低标准为：

- [ ] 页面、产品、案例、行业方案都严格执行“草稿 → 预览 → 发布 → 回滚”。
- [ ] 同一资源同一时刻只有一名管理员能编辑；不持锁不能保存、发布、回滚或下线。
- [ ] 普通用户与 Portal 接口无法看到编辑入口、草稿、锁、预览 Token、版本、审计或内部配置。
- [ ] 区块新增与排序仅限服务端注册组件和合法插槽，区块数量可扩充。
- [ ] 三类详情均有独立编辑器所需的完整后端生命周期，而非直接复用传统 CRUD。
- [ ] 发布、回滚、下线、删除均完成引用校验和缓存失效，且失败可追踪。
- [ ] 接口文档、错误码、数据库说明、迁移脚本和自动化测试同步交付。
