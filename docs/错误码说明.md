# 错误码说明

本文档只定义官网后台系统与 Portal 前台接口的数字错误码设计规则，不预先枚举每一个具体错误码取值。具体错误码由编码实现时按模块、业务场景和接口文档逐项设置，并通过枚举类统一维护。

错误码用于前端提示、接口联调、日志检索、测试断言和接口文档维护。新增或修改接口时，必须在对应接口文档中记录实际可能返回的错误码。

---

## 一、统一响应结构

所有接口统一返回以下结构：

```json
{
  "code": 0,
  "message": "操作结果说明",
  "data": {},
  "traceId": "20260620120000abcdef"
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `code` | `Integer` | 是 | 业务错误码。成功、失败都必须返回稳定数字编码，具体值在编码实现时定义。 |
| `message` | `String` | 是 | 面向调用方的简短提示。 |
| `data` | `Object` | 否 | 业务数据，失败时通常为空对象或 `null`。 |
| `traceId` | `String` | 是 | 请求追踪 ID，必须与日志中的 traceId 一致。 |

HTTP 状态码仍按 Web 语义返回，例如参数错误可返回 `400`，未登录可返回 `401`，无权限可返回 `403`，服务端异常可返回 `500`。业务判断以响应体中的 `code` 为准。

---

## 二、数字编码规则

错误码统一使用五位整数，按模块分段：

```text
MNNSS
```

含义：

* `MNN`：模块段，用于快速判断错误来源。
* `SS`：模块内序号，由编码实现时按业务场景递增分配。

示例：

```text
10000-19999  通用错误
20000-29999  认证与权限错误
30000-39999  站点、导航、首页与 AI 能力错误
40000-49999  产品模块错误
50000-59999  行业、案例与新闻错误
60000-69999  媒体资源错误
70000-79999  客户线索错误
80000-89999  审计、缓存与系统支撑错误
90000-99999  预留扩展或第三方集成错误
```

编码规则：

* 成功码固定使用 `0`。
* 业务失败码不得使用 `0`。
* 错误码一经对外使用，不得改号、删除或复用为其他含义。
* 同一模块内按业务场景分配序号，避免多个错误共用同一个码。
* 新增模块时必须先在本文档补充分段，再在代码枚举中定义具体错误码。
* 错误码不得携带用户输入、数据库 ID、手机号、邮箱、Token、密钥等动态或敏感信息。

---

## 三、模块分段建议

| 编码范围 | 模块 | 适用范围 |
| --- | --- | --- |
| `0` | 成功 | 请求成功。 |
| `10000-19999` | 通用 | 参数、分页、资源、状态、并发、重复数据等公共错误。 |
| `20000-29999` | 认证权限 | 登录、Token、账号、角色、权限、CSRF。 |
| `30000-39999` | 站点展示 | 站点配置、导航菜单、首页 Banner、CTA、量化指标、AI 能力卡片。 |
| `40000-49999` | 产品 | 产品分类、标签、产品、功能点、成效指标。 |
| `50000-59999` | 内容文章 | 行业场景、客户标签、新闻分类、新闻文章、富文本。 |
| `40200-40299` | 标杆案例 | 标杆案例卡片管理。 |
| `60000-69999` | 媒体 | 文件上传、校验、存储、处理、绑定、解绑、清理。 |
| `70000-79999` | 线索 | 客户咨询、处理状态、敏感信息查看、导出。 |
| `80000-89999` | 系统支撑 | 审计日志、缓存失效、配置、外部依赖、未知异常。 |
| `90000-99999` | 扩展预留 | 第三方集成、后续独立模块。 |

## 四、错误码枚举类规则

后端实现时应定义统一错误码枚举类，集中维护数字码、默认消息和模块信息。建议结构如下：

```java
public enum ErrorCode {
    SUCCESS(0, "COMMON", "操作成功"),

    // 具体错误码在编码实现时按模块分段补充。
    // 示例格式：PARAM_INVALID(10001, "COMMON", "请求参数不正确");

    ;

    private final int code;
    private final String module;
    private final String defaultMessage;

    ErrorCode(int code, String module, String defaultMessage) {
        this.code = code;
        this.module = module;
        this.defaultMessage = defaultMessage;
    }

    public int getCode() {
        return code;
    }

    public String getModule() {
        return module;
    }

    public String getDefaultMessage() {
        return defaultMessage;
    }
}
```

实现要求：

* 所有业务异常必须携带 `ErrorCode`，不得只携带裸数字。
* `ApiResponse.code` 对外返回 `ErrorCode#getCode()`。
* `message` 默认使用 `ErrorCode#getDefaultMessage()`，必要时可由业务异常传入更具体但安全的提示。
* 不允许在 Controller、Service、Mapper 中散落裸数字错误码。
* 不允许多个枚举项使用同一个数字码。
* 不允许枚举项名称含义与数字分段所属模块不一致。

## 五、错误码分层规则

错误码应按以下层次设计：

* **通用错误码**：适用于多个模块的参数错误、资源不存在、状态冲突、重复数据、分页错误、请求过频等场景。
* **认证权限错误码**：适用于未登录、登录失败、账号停用、账号锁定、Token 失效、角色无权、CSRF 校验失败等场景。
* **业务模块错误码**：适用于具体模块的唯一性冲突、关联资源不存在、内容状态不允许、媒体绑定不合法、富文本不安全等场景。
* **系统支撑错误码**：适用于缓存失效失败、审计写入失败、外部依赖不可用、对象存储失败、系统未知异常等场景。

设计原则：

* 同一错误在多个模块含义一致时，优先使用通用错误码。
* 错误需要前端展示不同提示、触发不同交互或测试需要精确断言时，应定义模块级错误码。
* 不为每一个字段创建过细错误码；字段级错误详情可放入 `data.fieldErrors`。
* 不为同一个业务语义定义多个等价错误码。

---

## 六、错误消息规则

`message` 是给调用方展示或调试的简短说明，应遵守以下规则：

* 语义清晰，能帮助前端或用户理解下一步动作。
* 不暴露 SQL、堆栈、服务器路径、类名、配置键、对象存储路径等内部细节。
* 不暴露 Token、密钥、验证码、完整手机号、完整邮箱、客户留言全文等敏感信息。
* 登录失败提示不得区分“账号不存在”和“密码错误”，避免账号枚举。
* 未授权访问资源时，可按安全需要返回无权限或资源不存在，避免泄露资源存在性。
* 系统未知异常应返回通用提示，详细原因只写入服务端日志。

---

## 七、字段级错误规则

参数校验失败时，响应可以在 `data` 中返回字段级错误信息：

```json
{
  "code": 10001,
  "message": "请求参数不正确",
  "data": {
    "fieldErrors": [
      {
        "field": "title",
        "message": "标题不能为空"
      }
    ]
  },
  "traceId": "20260620120000abcdef"
}
```

字段级错误规则：

* `field` 使用请求 DTO 字段名，不使用数据库字段名。
* `message` 描述字段问题，不输出正则表达式、数据库约束名或内部类名。
* 多字段错误可以一次性返回，便于前端集中提示。
* 字段级错误不替代业务错误码，顶层 `code` 仍必须稳定。

---

## 八、代码实现规则

* 错误码必须定义在统一枚举类中，不得在业务代码中散落裸数字。
* Controller 不直接拼接错误响应，应由统一异常处理器转换为统一响应结构。
* 可预期业务失败必须抛出统一 `BusinessException` 或等价业务异常，并携带 `ErrorCode`。
* 数据库唯一约束、乐观锁、外部依赖失败等底层异常必须转换为业务可理解的错误码。
* 未知异常统一转换为系统级错误码，并在服务端日志记录完整堆栈和 `traceId`。
* 错误码变更属于接口契约变更，必须同步更新接口文档和相关测试。

---

## 九、接口文档记录规则

每个接口条目必须记录该接口可能返回的错误码：

```markdown
* **错误码**：
    * `10001`：说明触发该错误码的业务场景。
```

记录要求：

* 只记录该接口实际可能返回的错误码，不罗列无关模块错误码。
* 通用错误码和模块级错误码都要记录。
* 如果接口新增错误场景，必须同步更新接口文档。
* 如果错误码被删除、改号或语义改变，必须评估前端、测试和调用方影响。

---

## 十、测试与 Review 规则

* 接口测试必须断言失败场景的 `code`，不能只断言 HTTP 状态码。
* 参数错误、未登录、无权限、资源不存在、状态冲突、重复数据等高频错误必须有测试覆盖。
* 安全相关错误必须验证响应中不包含敏感信息或内部实现细节。
* Review 时必须检查错误码是否集中定义、数字分段是否正确、接口文档是否同步。
* 不允许为了快速合并使用临时错误码或复用语义不匹配的错误码。
