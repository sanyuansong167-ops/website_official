# 云台数据官网前端实现说明

> 文档状态：设计与实施基线  
> 适用范围：官网 Portal 前台、预约交流表单，以及与后台内容管理和页面构建器的对接约束。  
> 参考来源：`picture/` 页面参考图、`docs/接口文档.md`、`docs/重构接口文档.md`。

## 1. 目标与边界

### 1.1 建设目标

官网前端应展示云台数据的企业能力、产品矩阵、行业方案、标杆案例、技术创新与联系入口，并满足以下目标：

- 将后台维护的内容稳定渲染为 SEO 友好、加载快速的公共官网页面。
- 视觉还原 `picture/` 中的品牌风格：科技感、留白、清晰的信息层级、轻量卡片与蓝色品牌强调。
- 保持内容数据驱动。页面不得将可运营的标题、文案、卡片、图片地址或联系方式硬编码为唯一来源。
- 为后续页面构建器提供可配置的区块容器和组件渲染能力，不把页面结构逻辑散落在业务页面中。
- 提供安全、可用、可观测的预约交流表单提交体验。

### 1.2 本期范围

本说明以 Portal 官网为主，包含：

1. 首页与公共导航、页脚；
2. 企业实力、AI 战略、核心能力、产品矩阵、行业解决方案、标杆案例、关于我们、联系我们页面或区块；
3. 产品/案例列表与详情跳转；
4. 匿名预约交流提交；
5. Portal API 的请求、缓存、错误与加载状态；
6. 管理后台对接所需的通用约束，不定义 Admin 页面完整交互稿。

不在本说明中实现或假定以下能力：独立登录注册、支付、在线客服、未在接口文档中定义的筛选接口、草稿的匿名直连访问权限策略。

## 2. 前端架构建议

### 2.1 推荐技术选型

前端工程尚未创建。建议新建独立前端应用，采用以下基线：

| 分类 | 建议 | 原因 |
| --- | --- | --- |
| 框架 | Next.js（React + TypeScript） | 适合官网的 SSR/SSG、路由、SEO 与首屏性能。 |
| 样式 | CSS Modules 或 Tailwind CSS + 设计 Token | 易维护、可组件化，并能控制响应式细节。 |
| 数据请求 | TanStack Query + 原生 `fetch` 封装 | 支持请求去重、重试控制、错误状态与失效。 |
| 表单 | React Hook Form + Zod | 保证预约表单的声明式校验与可访问性。 |
| 图标 | 统一图标库或受控 SVG 图标组件 | 保持线性图标、尺寸与颜色一致。 |
| 测试 | Vitest + React Testing Library；E2E 使用 Playwright | 覆盖组件、表单和关键导航路径。 |

若团队选择 Vue/Nuxt，也必须保持本文定义的路由、组件边界、接口数据模型、状态处理与验收要求。

### 2.3 数据边界

- **Portal 前端只调用 `/portal/api/**`**。不得使用 Admin API 获取公开内容。
- 文本、卡片、展示图片、排序和可见性由后端处理；前端不得根据管理端状态再次猜测内容是否应展示。
- 前端使用独立的 ViewModel 适配 API 响应，避免组件直接依赖数据库实体或后台表单 DTO。
- 后端公共接口已采用缓存。前端可做短时请求去重，但不得以超长客户端缓存掩盖后台内容更新。

## 3. 信息架构与路由

### 3.1 一级导航

参考首页导航与接口约定，建议默认路由如下。最终菜单名称、显示状态、链接类型和排序以 `GET /portal/api/site/navigation/menus` 返回为准。

| 导航 | 建议路径 | 主要内容 |
| --- | --- | --- |
| 首页 | `/` | Hero、核心数据、重点能力、产品、行业、案例与 CTA。 |
| 企业实力 | `/strength` | 资质荣誉、客户 Logo、企业实力指标。 |
| AI 战略 | `/ai-strategy` | 战略文案与 AI 能力卡。 |
| 核心能力 | `/capabilities` | 三大能力分类与子项。 |
| 产品矩阵 | `/products` | 产品卡片、状态标签、产品详情入口。 |
| 解决方案 | `/solutions` | 行业方案卡片与典型客户标签。 |
| 标杆案例 | `/cases` | 案例卡片墙、案例详情入口。 |
| 关于我们 | `/about` | 产学研、研发方向、发展历程、价值观、承诺。 |
| 联系我们 | `/contact` | 联系方式、合作方向和预约交流表单。 |

导航目标需兼容接口定义的 `GROUP`、`INTERNAL_ROUTE`、`PAGE_ANCHOR`、`EXTERNAL_LINK` 四种类型：

- `GROUP`：只展开二级菜单，不生成无意义的跳转。
- `INTERNAL_ROUTE`：使用站内路由，并保证 `routePath` 以 `/` 开头。
- `PAGE_ANCHOR`：跳转到页面锚点，需处理固定顶部导航造成的偏移。
- `EXTERNAL_LINK`：只接受后端返回的合法 `http/https` 地址；`openInNewTab=true` 时使用新标签页并设置 `rel="noreferrer noopener"`。

### 3.2 页面标题与 SEO

- 每页默认继承站点配置中的 `siteTitle`、`seoKeywords`、`seoDescription`。
- 页面构建器路由先调用 `GET /portal/api/page-builder/pages?routePath={path}` 获取 `pageKey`，再调用 `GET /portal/api/page-builder/pages/{pageKey}/meta` 获取元数据；未配置时才使用站点默认值。
- 产品和案例详情应使用其自身 `seoTitle`、`seoDescription`（接口存在时），否则回退到标题和摘要。
- 路由变更应设置 `title`、`description`、canonical URL、Open Graph 基础字段；禁止把含个人联系方式的线索数据写入页面元数据或埋点。

## 4. 视觉与交互规范

### 4.1 品牌视觉

参考图显示的视觉基线如下：

- **主色**：高亮品牌蓝，建议 Token `--color-brand-600: #2F63E9`；悬停使用更深蓝，浅色背景使用低饱和蓝灰。
- **深色区块**：用于 AI 战略、创新研发体系、页脚，建议底色为深海军蓝，文字为接近白色，卡片为半透明或深蓝灰。
- **中性色**：主文字深蓝黑、次文字蓝灰、边框淡蓝灰；不要使用纯黑大面积文本。
- **卡片**：白色或深色半透明底、`12–16px` 圆角、细边框、轻阴影；可点击卡片需具备悬停上浮和焦点可见状态。
- **图标**：图标容器为圆角方形，常用蓝/绿/橙/紫/粉作为模块识别色；同一区块内保持尺寸一致。
- **标题**：顶部分区小标签 + 大号粗体中文标题 + 可选副标题，居中或按 Hero 左对齐。

### 4.2 布局 Token

| Token | 建议值 | 说明 |
| --- | --- | --- |
| 内容最大宽度 | `1200px` | 桌面端主体容器。 |
| 桌面左右内边距 | `24px` | 宽屏下的最小安全留白。 |
| 移动端内边距 | `16px` | 小屏内容安全区域。 |
| 区块上下间距 | `80–112px` | 桌面端模块节奏。 |
| 卡片间距 | `16–24px` | 网格内一致使用。 |
| 正文行高 | `1.7–1.8` | 保证中文段落阅读性。 |
| 过渡时长 | `160–240ms` | 用于 hover、抽屉和按钮反馈。 |

### 4.3 响应式断点

| 视口 | 行为 |
| --- | --- |
| `>= 1200px` | 默认桌面栅格，产品 3 列、案例 3 列、行业 2 列、能力 3 列。 |
| `768px–1199px` | 导航允许收缩，卡片通常为 2 列，Hero 图文可保持双列。 |
| `< 768px` | 顶部菜单切换为抽屉；卡片单列；Hero、联系区、页脚列纵向排列；按钮易触达。 |

移动端不得仅缩小桌面页面：需保证正文不小于 `14px`，主要按钮高度不低于 `44px`，横向标签可换行或横滑，不能产生正文区域横向溢出。

### 4.4 可访问性

- 使用语义化 `header`、`nav`、`main`、`section`、`footer`、`button`、`form`、`label`。
- 菜单抽屉、模态框、Toast 必须支持键盘、焦点管理和 Escape 关闭。
- 装饰性图标使用空 `alt` 或 `aria-hidden`；业务图片提供有意义的替代文本。
- 文字与背景的对比度达到 WCAG AA；仅凭颜色区分的状态必须有文字或图标补充。
- 所有错误提示与异步状态应通过 `aria-live` 可被读屏识别。

## 5. Portal 页面与组件说明

### 5.1 全局布局

#### 顶部导航 `SiteHeader`

- 桌面端：左侧品牌 Logo，中部一级菜单，右侧主 CTA“预约交流”。
- 当前路由高亮以浅蓝圆角底或文字色区分；一级菜单含子项时支持鼠标、键盘展开。
- 滚动后采用带模糊或白色背景的固定导航，但必须保持内容对齐与可读性。
- Logo 使用站点配置的亮/暗资源；加载失败时显示文本品牌，不展示破损图片标识。

#### 页脚 `SiteFooter`

- 深色背景，包含品牌简介、快捷链接、服务领域、联系方式、社交入口、隐私政策、服务条款和备案号。
- 快捷链接和服务领域应由导航与站点配置派生；没有数据时隐藏对应链接，不保留空列表。
- 对外社交链接必须使用受控 URL，外链开启新标签页的安全属性。

### 5.2 首页 `/`

首页采用模块化区块组合，支持由页面构建器替换或编排。默认区块建议如下：

1. **Hero 首屏**：
   - 左侧为认证标签、主标题、介绍文案、主/次 CTA 与四项数据摘要；右侧为 Banner 主图或动态视觉。
   - 数据来自 `GET /portal/api/home/banner`、`GET /portal/api/home/metrics` 和站点配置；当 Banner 无图片时保持文案布局，不使用空白图片框。
2. **企业实力摘要**：
   - 资质与荣誉、服务客户、统计指标，视觉参照 `picture/企业实力.png`。
   - 使用 `GET /portal/api/honors`、`GET /portal/api/client-logos`、`GET /portal/api/strength-metrics`。
3. **AI 战略摘要**：
   - 深色区块，标题、战略文案、四张能力卡及 CTA，视觉参照 `picture/AI战略.png`。
   - 使用 `GET /portal/api/ai-cards`。
4. **核心能力摘要**：
   - 三列能力分类卡，显示分类说明及子项勾选列表，视觉参照 `picture/核心能力.png`。
   - 使用 `GET /portal/api/capabilities`；子项过多时默认展示前 5 项并提供进入详情页的入口。
5. **产品矩阵摘要**：
   - 三列产品卡，展示 Icon、名称、二级定位、摘要、状态标签与详情链接，视觉参照 `picture/产品矩阵.png`。
   - 使用 `GET /portal/api/products`。
6. **行业解决方案摘要**：
   - 两列行业卡，展示名称、Icon、描述与典型客户标签，视觉参照 `picture/解决方案.png`。
   - 使用 `GET /portal/api/industry-solutions`；接口尚未提供时区块使用显式占位并列为联调阻塞项，不能硬编码替代正式数据。
7. **标杆案例摘要**：
   - 三列案例卡片，展示 Icon、标题、描述、标签和箭头入口，视觉参照 `picture/标杆案例.png`。
   - 使用 `GET /portal/api/cases`；最多展示 6 条，按接口排序。
8. **联系 CTA**：跳转至 `/contact` 或联系锚点。

### 5.3 企业实力 `/strength`

- 顶部使用“十余年的行业积累，是云台数据持续创新的基础”类标题结构。
- 资质荣誉使用可读的文本卡片，不只展示奖杯图标；服务客户使用 Logo 墙，图片需有客户名称替代文本。
- 底部核心指标以图标、数字和单位/说明呈现；数字动画仅在用户未开启减少动态效果时执行。

### 5.4 AI 战略 `/ai-strategy`

- 深色背景，左侧战略导语，右侧或下方四宫格 AI 卡片。
- 每张卡包含 `icon`、英文小标题（可选）、中文标题、摘要、详情入口。
- 主 CTA 跳转到联系页并携带合作方向上下文，例如 `?direction=AI应用开发`；不得在 URL 中携带姓名、邮箱、电话等个人数据。

### 5.5 核心能力 `/capabilities`

- 将分类和子项渲染为三栏或响应式网格。
- 分类卡含图标、名称、说明与子项；子项是展示信息，不应使用无语义的可点击勾选控件。
- 返回结果为空时不显示空白大区块，展示简洁空状态并保留联系 CTA。

### 5.6 产品矩阵 `/products` 与详情

- 列表卡片显示产品状态。状态文案“已发布”使用灰蓝标签，“即将发布”使用粉色高亮；未知状态使用中性标签，不通过字符串猜测业务权限。
- 已发布产品可进入 `/products/{id}`；即将发布产品默认不可进入详情，除非 API 返回允许访问的详情信息。
- 详情页展示名称、Logo、定位、内容、图片、SEO 信息及推荐/关联内容（字段由接口实际返回决定）。

### 5.7 行业解决方案 `/solutions`

- 使用两列大卡片，底部以低饱和色块承载“典型客户”标签。
- 标签不可被当作筛选条件，除非后端明确定义对应查询参数。
- 点击整张卡或箭头后进入可用的方案详情或联系咨询；无详情路由时使用无障碍按钮跳转至联系页。

### 5.8 标杆案例 `/cases` 与详情

- 列表默认三列栅格、移动端单列；单卡需要标题、摘要、最多三枚标签、图标和明确的详情入口。
- 详情页请求 `GET /portal/api/cases/{id}`，展示 `title`、`customerName`、`industry`、`background`、`solution`、`result`、`content`、封面和图集，以及关联产品/推荐案例（若响应包含）。
- 请求 404 或内容不可见时显示统一“不存在或已下线”页面，并提供返回案例列表的按钮。

### 5.9 关于我们 `/about`

页面按以下区块排列：

1. **创新研发体系**：深色区块；合作高校 Logo 卡与重点研发方向卡，参考 `picture/行业.png`。数据来自 `GET /portal/api/partner-universities`、`GET /portal/api/research-directions`。
2. **发展历程**：中央垂直时间轴，桌面端左右交错、移动端单列靠左，参考 `picture/发展历程，时间线.png`。数据来自 `GET /portal/api/timeline-events`，按接口顺序直接渲染。
3. **核心价值观与承诺**：三张价值观卡 + 浅蓝背景承诺区，参考 `picture/核心价值.png`。数据来自 `GET /portal/api/value-cards` 与“我们的承诺”聚合接口。

### 5.10 联系我们 `/contact`

- 左侧上方：地址、商务电话、邮箱，使用 `GET /portal/api/contact-info`。
- 左侧下方：蓝色合作方向区域，使用 `GET /portal/api/cooperation-direction-tags`；标签可点击以预填预约表单的合作方向，但必须允许用户修改。
- 右侧：预约交流表单，参考 `picture/联系我们.png`。
- 表单字段：

| 字段 | 前端规则 |
| --- | --- |
| 姓名 | 必填，去除首尾空白，限制合理长度。 |
| 公司 | 必填，去除首尾空白，限制合理长度。 |
| 邮箱 | 必填，使用标准邮箱格式校验。 |
| 电话 | 默认选填；如后端配置为必填，则在配置明确下标记必填。 |
| 需求描述 | 选填或按后端规则必填，限制长度；按纯文本提交。 |
| 合作方向 | 推荐从标签预填；具体字段名必须以接口文档为准。 |

提交成功后清空表单、保留选中的合作方向或展示确认信息；失败时保留用户输入并显示不包含敏感回显的错误提示。禁止把完整表单内容写入浏览器日志、埋点或 `localStorage`。

## 6. 接口对接规范

### 6.1 HTTP 客户端

- 统一从环境变量读取 API 基地址，例如 `NEXT_PUBLIC_PORTAL_API_BASE_URL`；禁止在组件中拼接服务域名。
- 所有响应经过 `unwrapApiResponse` 统一处理，生成包含 `code`、`message`、`requestId`（如有）的前端错误对象。
- Portal GET 请求默认不携带身份凭证；表单提交按后端跨域与防刷规则发送必要请求头。
- 请求超时、网络错误、5xx、业务错误分别处理。不要把后端错误消息原样暴露给用户，尤其是安全或内部异常信息。
- 所有 URL 路径参数使用 `encodeURIComponent`；任何富文本渲染前必须经过白名单净化。

### 6.2 已定义 Portal 接口映射

| 模块 | 接口 | 前端用途 |
| --- | --- | --- |
| 站点 | `GET /portal/api/site/config` | 标题、SEO、Logo、品牌文案。 |

接口路径、请求参数、响应字段与错误码以 `docs/接口文档.md` 为唯一详细依据；页面构建器相关内容同时以 `docs/重构接口文档.md` 为准。若实现与文档不一致，应先修正文档和后端契约，再修改前端，不允许前端静默兼容多个未声明字段。

### 6.3 缓存与刷新

- Portal GET 接口由后端 Redis 缓存；后台写操作在事务提交后主动失效对应键并延迟二次删除。
- 前端查询缓存建议：站点配置与导航 `5 min`，列表数据 `1–5 min`，但页面重新获得焦点或关键路由进入时允许后台刷新。
- 不对线索提交、认证、预览 Token 等接口做客户端持久缓存。
- 当使用 SSR/SSG 时，页面再验证时间必须不大于后端内容更新可接受时延，并提供按路由或标签再验证的扩展位。

## 7. 页面构建器渲染约定

### 7.1 渲染链路

1. 根据当前路径请求 `GET /portal/api/page-builder/pages?routePath={path}`，读取渲染 Schema 与响应中的 `pageKey`。
2. 使用 `pageKey` 请求 `GET /portal/api/page-builder/pages/{pageKey}/meta` 获取页面元数据。
3. 校验 Schema 的版本、组件类型和必填配置。
4. 使用受控组件注册表将 `componentType` 映射到前端组件。
5. 各区块仅接收明确的配置对象与数据源，不执行任意脚本，不解析任意 HTML。
6. 无法识别的组件显示开发环境诊断信息；生产环境记录受控错误并跳过该区块，保证其他页面内容继续展示。

### 7.2 受控组件注册表

首批建议支持：`HeroBanner`、`MetricGrid`、`CardGrid`、`LogoWall`、`CapabilityColumns`、`Timeline`、`RichText`、`ContactForm`、`CtaBanner`。

- 组件类型使用前端白名单，不能由后台下发模块路径、JavaScript 或 CSS 文本。
- 配置中的 URL、颜色、图片、富文本均需按字段类型校验；图片必须来自受控媒体 URL。
- 草稿预览只可使用后端签发的受控 Token，Token 不得存入 URL 以外的持久化存储，不得写入日志。

## 8. 状态、异常与安全处理

### 8.1 通用状态

| 状态 | 要求 |
| --- | --- |
| 初始加载 | 使用与目标模块尺寸接近的骨架屏，避免布局抖动。 |
| 空数据 | 隐藏无意义区块或显示简洁空状态，不展示技术错误。 |
| 请求失败 | 展示可重试提示；保留上一次可用数据时标明刷新失败。 |
| 404 | 使用统一不存在页面，提供返回首页或列表入口。 |
| 5xx/网络异常 | 提示“服务暂不可用，请稍后重试”，并保留重试按钮。 |
| 速率限制 | 明确提示操作过于频繁并显示可重试时间（若后端返回）。 |

### 8.2 预约表单安全

- 客户端校验只改善体验，不能替代服务端校验、IP 限流、敏感词/反垃圾策略。
- 防止重复提交：提交中禁用按钮；请求完成后恢复。不得因为按钮禁用而忽略服务端幂等与限流响应。
- 文本作为纯文本提交；如果未来支持富文本，必须使用白名单净化并进行安全评审。
- 不在 URL、页面标题、错误上报上下文或分析平台中记录姓名、邮箱、电话、需求描述。
- 失败提示只说明可行动信息，不展示数据库、栈追踪、权限或防刷策略细节。

## 9. 性能要求

- Hero 中的主要视觉可使用优化后的 WebP/AVIF，并声明尺寸以减少 CLS；非首屏图片懒加载。
- 页面首屏只加载必要字体和图标；避免整包引入大图标库。
- 下方模块可按路由或可视区动态加载，但不得造成可见内容闪烁。
- 图片均使用媒体接口返回的受控 URL；对图片加载失败提供品牌占位样式。
- 避免在客户端重复请求 Header、Footer 与首页区块已获得的数据；通过服务端数据预取或请求缓存去重。
- 基线目标：LCP `< 2.5s`、CLS `< 0.1`、INP `< 200ms`（以生产真实用户监控为准）。

## 11. 实施顺序

1. 初始化前端工程、环境变量、全局 Token、HTTP 客户端、错误边界和基础布局。

## 12. 联调前确认项

以下事项应在创建正式前端功能前与后端确认，避免前端基于推测实现：

1. 前端工程框架与部署方式（SSR/SSG/纯 SPA）及 API 反向代理规则。
2. Portal API 响应统一包装格式、分页字段、`requestId` 与 404 的业务错误码。
3. 产品详情、行业方案详情、承诺聚合接口的最终路径和字段是否已实现。
4. `POST /portal/api/leads` 的确切请求体、是否要求合作方向、电话必填配置和成功响应语义。
5. 媒体 URL 域名、跨域策略、图片处理规格与下载/预览权限。
6. 页面构建器 Schema 的版本、组件类型、配置 JSON 规范与预览 Token 传递方式。
7. SEO 域名、备案号、隐私政策与服务条款的正式内容。

在上述事项未确认前，前端可以完成组件、布局、Mock 数据和适配层，但不能把推测字段作为生产契约。
